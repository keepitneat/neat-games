## Strengths

The pure logic / DOM separation is tight and pays off — the test suite (22 new tests, all green) actually exercises real behaviors: merge-at-most-once, four-equal-tile double-merge, full-board spawn guard, the checkerboard game-over. `slideLine` is elegant and correctly handles every edge case in its traversal. The move-plan design (id-keyed node animation, consumed tiles tracked separately) is the right call for smooth animation without teleporting tiles. The win+dead-board edge case (pressing "keep going" onto a dead board) is correctly handled in `hideOverlay`. Encoding is clean — no binary files in the diff, no NUL bytes.

## Critical

**1. `2048/app.js:136-137` — `persist()` runs before `checkEnd()`, so the win state is never saved on the frame it fires.**
`finalize()` calls `persist()` then `checkEnd()`. `checkEnd()` is what sets `state.won = true`. So the game is saved with `won: false` on the frame the player reaches 2048. On reload, `boot()` restores `won: false`, and the next successful move triggers `checkEnd()` again — the win banner fires a second time.
Fix: swap the call order — `checkEnd()` before `persist()` — so `won` is `true` in the save file. (Or call `persist()` again inside `checkEnd` when `won` flips.)

## Important

**1. `2048/app.js:275` — resume path calls `checkEnd()` when `!canMove`, but doesn't check `state.won` before doing so.** Mostly harmless (`checkEnd` skips the win branch when `won` is true) but it's a redundant `canMove` check and leaks `checkEnd`'s internals into the resume path. A `showGameOver()` helper both callers share would be cleaner.

**2. `2048/logic.test.js` — `move('down', ...)` is not tested; three of four directions are covered.** Down is `rows:false, rev:true` — the one untested combination. A symmetric `rev` bug would be invisible. A single "column of two 2s merges at the bottom" test closes it.

**3. `2048/app.js:120-121` — `survivor.firstChild` accessed twice before being assigned to a variable.** Correct as written (same node, synchronous), but assign `const inner = survivor.firstChild` first, then use it — it'll silently break if a whitespace text node is ever prepended to `.tile`.

## Minor

**1. `2048/app.js:26` — `rng: makeRng(1)` in the initial state is a dead value** (overwritten by `startNew`/resume before any spawn). Initialize to `null` or drop the stub.

**2. Theme logic is hand-rolled rather than delegated to `shared/theme.js`**, which sits in the SW shell unused by either app. Pre-existing (Crowns does the same); target for a cleanup pass, not a blocker.

**3. `2048/index.html:32` — second `<div class="left">` actually holds the right-hand control.** Consistent with crowns; inherited label debt.

## Nits

1. `persist()` doesn't save `state.over` (intentional — re-derived from `canMove` at boot); a one-line comment would make the omission clearly deliberate.
2. `prefers-reduced-motion` JS check (`app.js:12`) is a one-time read; won't react to a mid-session OS change (CSS still wins). Same as crowns; acceptable.
3. No test for the resume-with-a-2048-tile-and-`won:false` scenario (the state that exposes the Critical bug). Add once the bug is fixed.

## Verdict

**FIX-THEN-SHIP.** Must-fix before shipping: swap `persist()`/`checkEnd()` in `finalize()` so the persisted `won` state is correct. Everything else is minor/nit-level. The logic is sound, the animation wiring is correct, the test suite is real, the win+dead-board edge case is handled, the PWA shell is complete, and the NEAT thesis holds.
