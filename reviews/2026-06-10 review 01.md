# Staff Review — Minesweeper

**Scope:** the `minesweeper` branch diff vs `main` at the time of review (game #3 for the Neat Games hub: `minesweeper/` engine + UI, plus hub `index.html` and `service-worker.js` edits).

---

The engine is genuinely clean — pure, injected rng, first-click-safe done right, and the test suite (26 passing) is well-constructed with an *independent* neighbor oracle and hand-built fixtures rather than trusting the engine against itself. Approach B is implemented consistently. The bulk of what follows is in `app.js` (the untested DOM layer, as expected) and a couple of CSS/token slips.

---

## 🔴 Must-fix

1. **`minesweeper/app.js:219` — first-reveal-that-instantly-wins never starts, and a normal first reveal mis-gates the timer.** In `doReveal`, the guard is `if (res.state.status !== 'new' && ui.startedAt === null) startTimer()`. But `reveal()` on a fresh board calls `placeMines`, which sets status to `'playing'` (or `'won'`/`'lost'`), so it's never `'new'` after a reveal — the `!== 'new'` check is always true and is dead. That's harmless *here*, but the real problem is ordering: `startTimer()` runs, then `applyResult(res)` runs and *also* calls `startTimer()` when status is `'playing'`. On a first click that **instantly wins** (geometrically rare but reachable), `applyResult` takes the win branch and `stopTimer()`s with `startedAt` set one line earlier, so the recorded time is ~0ms and becomes an unbeatable "best." Collapse timer ownership into `applyResult` (see Discussion).

2. **`minesweeper/minesweeper.css:120` — `var(--text)` is an undefined token; the design system uses `--fg`.** `styles.css` defines `--fg`, `--muted`, `--card`, `--accent` — there is no `--text`. So `.ms-cell { color: var(--text) }` resolves to inherited color with no fallback. It mostly hides because `[data-n]` rules override numbered cells and emoji render in their own glyph colors, but it's a real undefined reference to the design system and a dark-mode legibility hazard. Change to `var(--fg)` to match 2048/crowns.

3. **`minesweeper/persist.js` + `app.js:330` — a corrupt/partial saved game with `status === 'playing'` is restored without validating `cells`.** `boot` guards on `saved.status === 'playing' && DIFFICULTIES[saved.difficultyKey]`, then `startGame` does `buildBoard()` which iterates `ui.state.cells`. If the stored blob is `playing` but `cells` is missing/empty, `buildBoard` produces an empty board with a *running timer* and no way to act — a soft-lock. 2048's `boot` defends with `Array.isArray(saved.tiles) && saved.tiles.length`; mirror that (`Array.isArray(saved.cells) && saved.cells.length === saved.rows * saved.cols`).

## 🟡 Should-fix

1. **`minesweeper/app.js` — keyboard operability is incomplete for the core interaction.** Cells are real `<button>`s (Enter/Space reveal works, flag-mode + Enter flags, Enter on a number chords), but there's no arrow-key navigation across the `role="grid"`, no roving tabindex (a 16×16 board is 256 tab stops), and the board declares `role="grid"`/`role="gridcell"` without the row structure that pattern implies. Either implement minimal grid keyboard semantics (roving tabindex + arrows + flag key) or drop `role="grid"`/`gridcell` to plain buttons with positional labels so you're not promising a pattern you don't deliver. Make it a decision, not an accident.

2. **`minesweeper/minesweeper.css:134` — `.ms-cell.mine-hit` is dead; the clicked mine is never distinguished.** The CSS reserves a distinct red (`#e2675a`) for "the mine the player clicked," but `app.js` never adds `mine-hit`. On a loss every mine looks identical. Either wire it (engine tags the triggering id; `paintCell` adds `mine-hit`) or delete the rule.

3. **`minesweeper/app.js:120-143` — `startGame` does five jobs and the `restored` overload is the seam to split.** Resets difficulty, branches resume-vs-new (each with its own timer/rng/clear logic), then unconditionally rebuilds/paints/syncs. Suggest `resumeGame(restored)` + `newGameUI(difficulty)` each setting up state + timer, then a shared `renderGame()`.

4. **`minesweeper/app.js:160-187` — `endGame` re-reveals mines that `reveal()` already revealed.** On a loss, `reveal` already set every mine to `revealed` and `applyResult`→`repaint` already painted them. `endGame`'s loss branch re-maps + repaints. (Note: the repaint-all IS load-bearing for `flag-wrong` styling, but the cell re-map is redundant.)

5. **`minesweeper/app.js:107, 128-129` — the 250ms tick can disagree with recorded time at the win moment.** `setInterval(tickTimer, 250)` means the displayed second can lag the true second by up to 250ms; a player can see "1:23" and be told "New best 1:24." Minor honesty wrinkle; tick more often or accept it.

## Discussion

### The timer/first-reveal gate (`app.js:219`)
`if (res.state.status !== 'new' && ui.startedAt === null) startTimer()` reads as "start the clock once the game begins," but a fresh board is `'new'` only *before* the first reveal; `placeMines` flips it to `'playing'` immediately, so the `!== 'new'` check is always-true and dead. `applyResult`'s own `status === 'playing'` check does the real gating a line later. Collapse to a single owner: let `applyResult` start/stop the clock based on resulting status, set `startedAt` when `minesPlaced` first becomes true, and delete `doReveal`'s pre-start. Then the "instant win = 0ms best" edge can't exist regardless of board size.

### `role="grid"` without the grid (a11y)
Declaring `role="grid"` + `role="gridcell"` tells assistive tech "expect 2-D arrow-key navigation and a single tab stop," but the implementation gives 256 sequential tab stops and no arrow handling — a *worse* experience than plain buttons because the announced affordance doesn't match behavior. Two honest options: (a) implement the grid pattern properly (roving tabindex + arrows + flag key); or (b) downgrade to plain `<button>`s with positional `aria-label`s, accept Tab traversal. Given the mobile-first thesis, (b) is a legitimate ship-now choice with (a) as a follow-up ticket — but make it a decision.

### Undefined `--text` token
`var(--text)` with no fallback doesn't error — it falls through to inherited color, which happens to be `--fg` on `body`, so it *looks* correct today by accident. The moment a cell's inheritance chain changes, the numberless/emoji cells use a color nobody chose. The token-correctness bar in this codebase treats "references a token that doesn't exist" as a defect, not taste.

### What's done well
The engine's first-click-safety (excluding the clicked cell *and* its neighbors before shuffling) guarantees an opening cascade, and the test at `logic.test.js:56` actually asserts the cascade (`changed.length > 1`). The chord tests (`wrong flag → lose`, `can win by clearing the last safe cells`) are exactly the multi-cell interaction cases single-record fixtures miss. The service-worker bump to `v9` with all six minesweeper paths added to `APP_SHELL` is correct and complete.
